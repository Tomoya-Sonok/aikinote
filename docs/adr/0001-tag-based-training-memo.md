# タグごとの稽古記録メモのデータモデル（#280）

稽古記録ページに「タグごとにメモ入力」モードを追加するにあたり、メモ本体を別テーブル `TrainingPageMemo`、メモとタグの関連を `TrainingPageMemoTag`（1メモにつき1〜3タグ）として新設し、`TrainingPage` には入力モードを表す `content_mode`（`free` / `tag_based`）列を追加した。`tag_based` ページの `TrainingPage.content` は空文字で保存し、本文はメモ側に持つ。

## Considered Options

- **メモを別テーブル + メモ↔タグ中間テーブル（採用）**: 既存の `TrainingPageTag`（ページ↔タグ M:N）の流儀に揃い、1メモに複数タグ・1ページに複数メモを素直に表現できる。
- **`TrainingPageMemo.user_tag_ids` を配列カラムにする（不採用）**: テーブルは減るが、タグ削除時の追従や JOIN による取得が `TrainingPageTag` と別流儀になり、一貫性が崩れる。
- **`content_mode` 列を持たず、メモの有無でモードを推論（不採用）**: 「メモ0件の tag_based ページ」と「free ページ」を区別できず、表示・編集時の分岐が曖昧になる。

## Consequences

- ページ単位のタグ絞り込み（一覧フィルタ）を壊さないため、`tag_based` ページでも**全メモのタグの和集合**を従来どおり `TrainingPageTag` に保存する（派生データ。メモ保存・更新時に作り直す）。この二重保持は非自明なので注意。
- フリーワード検索は現状タイトルのみが対象（本文・メモ本文は未対象）。`tag_based` ページもタイトルは必須なので検索から漏れない。本文横断の全文検索が必要になったら別途対応する。
